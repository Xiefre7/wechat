### C2. 分享接收页直接修改 mockData，数据不持久化

**文件**：`pages/bank/share-receive/index.js:138-139`

**问题描述**：`handleImport` 方法中直接操作内存中的 mockData：
```js
mockData.banks.push(newBank);
newQuestions.forEach((q) => mockData.questions.push(q));
```
mockData 是 `require` 返回的内存对象，小程序冷启动后会被重新加载，导入的分享题库会完全丢失。用户导入后看似成功，但下次打开小程序题库消失，属于严重的数据丢失问题。

**修复建议**：应通过云函数将分享题库写入云数据库，或在本地使用 `wx.setStorageSync` 持久化用户导入的题库（独立于 mockData）。建议复用 `import-preview` 的 `quickstartFunctions` 云函数 `importBank` 类型。

### C3. 背题页"没记住"按钮缺失

**文件**：`pages/memorize/index.wxml`、`pages/memorize/index.js:192-203`

**问题描述**：JS 中定义了 `markNotMemorized` 方法用于标记"没记住"，但 WXML 中只有"记住了"按钮（第 70-73 行），没有"没记住"按钮。导致：
1. 用户无法标记未掌握的题目
2. `finishMemorizeSession` 统计 `notRememberedCount` 永远为 0

**修复建议**：在 `memorize/index.wxml` 的 `.memorize-actions` 区域增加"没记住"按钮：
```xml
<view class="memorize-actions">
  <button class="not-remember-btn" bindtap="markNotMemorized">没记住</button>
  <button class="remember-btn-large" bindtap="markMemorized">记住了</button>
</view>
```

### C4. 题库列表分享 onShareAppMessage 异步返回失效

**文件**：`pages/bank/list/index.js:424-503`

**问题描述**：`onShareAppMessage` 中调用 `this.prepareShareData(bank)` 并 `return` 其返回的 Promise。但微信小程序的 `onShareAppMessage` 是同步回调，不会等待 Promise resolve。实际行为：
- 当 `prepareShareData` 走本地 mockData 分支时，`doShare` 会调用 `wx.cloud.callFunction` 返回 Promise，分享卡片此时已用 undefined 内容发送
- 降级路径中 `wx.showModal` 是异步的，但分享卡片已经发出

**修复建议**：
1. 分享前预先调用云函数获取 shareCode，缓存后用于分享
2. 或改为点击分享按钮时先 `showLoading` + 云函数获取 shareCode，再手动调用分享
3. 降级方案中不应在 `onShareAppMessage` 内调用 `wx.showModal`

### C5. 斩题管理页深色模式实现方式与其他页面不一致

**文件**：`pages/slash/manage/index.wxml:1`、`pages/slash/manage/index.wxss:213-248`

**问题描述**：
1. WXML 第 1 行 `<view class="manage-page">` 未添加 `{{isDark ? 'theme-dark' : ''}}` 类
2. JS 中没有 `isDark` 字段和主题监听
3. WXSS 使用 `@media (prefers-color-scheme: dark)` 而非 `.theme-dark` 类选择器

后果：当用户在 App 内手动切换深色模式时（不跟随系统），此页面不响应；与其他页面主题切换行为不一致。

**修复建议**：与其他页面统一，JS 中添加 `isDark` 字段并在 `onLoad`/`onShow` 中读取 `app.globalData.effectiveTheme`，WXML 根节点加 `{{isDark ? 'theme-dark' : ''}}`，WXSS 改用 `.theme-dark` 类选择器替换 `@media`。

### C6. 错题复习 SM-2 重复更新

**文件**：`pages/wrong/review/index.js:334-377`（nextQuestion）、`pages/wrong/review/index.js:380-416`（finishReview）

**问题描述**：在 `nextQuestion` 中会调用 `wrongBook.reviewAnswer(lastAnswer.wrongId, lastAnswer.isCorrect)`（第 345 行）。当用户做完最后一题点击"完成复习"时，`nextQuestion` 检测到是最后一题，调用 `finishReview`，而 `finishReview` 第 387-389 行又会再次调用 `wrongBook.reviewAnswer` 处理同一道题。

后果：同一道题的 SM-2 间隔被推进两次，导致错题复习间隔异常加长，用户可能很久才能再次复习该错题。

**修复建议**：在 `finishReview` 中检查 `lastAnswer` 是否已处理，或统一在 `finishReview` 处理所有题目的 SM-2 更新，`nextQuestion` 中不再处理最后一题。例如：
```js
nextQuestion() {
  // ...
  if (lastAnswer && lastAnswer.isCorrect !== null && !lastAnswer._reviewed) {
    wrongBook.reviewAnswer(lastAnswer.wrongId, lastAnswer.isCorrect);
    lastAnswer._reviewed = true;
  }
  // ...
}
finishReview() {
  const lastAnswer = this._reviewAnswers[this._reviewAnswers.length - 1];
  if (lastAnswer && lastAnswer.isCorrect !== null && !lastAnswer._reviewed) {
    wrongBook.reviewAnswer(lastAnswer.wrongId, lastAnswer.isCorrect);
  }
  // ...
}
```

## 二、中等问题（Medium）

### M1. OCR 次数校验仅客户端，可被绕过

**文件**：`pages/bank/import/index.js:474-493`

**问题描述**：OCR 每日 20 次限制仅通过 `wx.getStorageSync('ocrUsage')` 客户端 Storage 实现，代码中 `// TODO: 后续接入云函数端 OCR 次数校验` 也明确标注。用户清除小程序缓存即可重置次数，可能导致云函数 OCR 成本失控。

**修复建议**：在云函数 `parseOCR` 中基于 openid + 日期做次数校验，使用云数据库或 Redis 记录当日调用次数。

---

### M2. 真题列表缺少加载失败提示和重试机制

**文件**：`pages/exam/list/index.js:43-47`、`pages/exam/list/index.wxml:54-64`

**问题描述**：`loadData` 的 catch 分支仅 `console.error`，WXML 只有"空状态"和"加载中"两种 UI，没有"加载失败"状态。弱网下用户看到的是空列表，无法区分"无真题"和"加载失败"。

**修复建议**：增加 `loadError` 状态字段，catch 中设置为 true，WXML 增加失败状态 UI 和"重试"按钮。

---

### M3. 历史记录跳转逻辑混乱

**文件**：`pages/history/index.js:64-72`

**问题描述**：`handleItemTap` 使用 `wx.switchTab` 跳转 `/pages/bank/list/index`，但 bank/list 是分包页面不是 tabBar 页面，`switchTab` 必然失败，再降级为 `navigateTo`。这种"先错后补"的写法逻辑不清晰，且点击历史记录卡片没有传递任何参数（如 bankId），用户无法直接继续练习该题库。

**修复建议**：
1. 直接使用 `wx.navigateTo` 跳转
2. 传递 `bankId` 参数，让 bank/list 定位到对应题库并支持"继续练习"

### M4. 预览页删除题目后 wx:key 重复风险

**文件**：`pages/bank/import-preview/index.js:220-235`

**问题描述**：`removeQuestion` 通过 `filter` 删除题目，但保留的题目 `_idx` 字段未重新编号。WXML 中 `wx:key="_idx"` 依赖此字段，删除中间题目后可能出现 key 重复或框架渲染异常（虽然实际 `_idx` 是原始索引不会重复，但删除后数组索引与 `_idx` 不再对应，后续 `toggleEdit` 等通过 `data-index` 操作时仍用数组索引，与 `_idx` 混用易出错）。

**修复建议**：删除后重新编号 `_idx`，或统一使用数组索引作为 key 并确保操作一致。

### M5. 分享接收页缺少再次转发能力

**文件**：`pages/bank/share-receive/index.js`

**问题描述**：页面未定义 `onShareAppMessage` 和 `onShareTimeline`，用户收到分享后无法再次转发给好友，限制了题库的传播能力。

**修复建议**：添加 `onShareAppMessage`，返回当前题库的分享信息（若已导入则用新 bankId 重新生成 shareCode）。

### M6. 分享按钮 padding-left hack 导致点击区域异常

**文件**：`pages/bank/list/index.wxss:270-279`

**问题描述**：
```css
.share-btn {
  padding-left: 190rpx !important;
  ...
  opacity: 0.5;
}
```
用 `padding-left: 190rpx` 撑开 button 触发区域以覆盖整个卡片右侧，但视觉上分享图标在右侧。这导致：
1. 按钮实际点击区域远大于视觉区域
2. `opacity: 0.5` 让按钮看起来不可点击
3. 可能与卡片整体的 `bindtap="toggleKnowledgePoints"` 点击区域重叠

**修复建议**：移除 padding hack，分享按钮独立定位，或使用 `button` 的 `open-type="share"` 配合正常的样式布局。
